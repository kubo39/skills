# API の人間工学 — 呼び出し地点から設計する

判断基準はただ一つ:**「利用者が自然に書いたコードが、正しく・速く・読みやすいか」**。
全原則を貫く思想は3つ — 呼び出し地点の自己文書化 / 自然な書き方 = 正しい書き方になるよう誘導
（pit of success）/ 利用者の書いた形を受け入れる。
レビューでのテスト:**「呼び出し地点だけ見てコードレビューができるか」**。

## 原則1: 引数の意味を呼び出し地点に現す

裸の `true`/`false`/`0`/`1` が並ぶ呼び出しを作らない。2値は名前付きフラグ型
（`No.caseSensitive` — 名前ごとに別型なので順序違いはコンパイルエラー）、3値以上は enum
（`SpanMode.depth`）。**理由**: コードは書かれる回数より読まれる回数が多い。
`indexOf(s, 'a', false)` を読んだ人は必ずドキュメントを引く。`No.caseSensitive` なら引かない。

## 原則2: 契約を名前に焼き込む — 体系として

個々の良い名前ではなく**規約の体系**を作り、全 API で貫く。Phobos の体系:

| 規約 | 意味 | 例 |
|---|---|---|
| -er 形（動作主名詞） | 遅延（ビューを返す） | `splitter`, `joiner` |
| 動詞そのもの | 即時（配列を返す/実行する） | `split`, `join`, `each` |
| `by*` | 遅延ビュー | `byLine`, `byCodeUnit` |
| `is*` / `has*` | 判定 trait / 能力 trait | `isInputRange`, `hasLength` |
| `assume*` | **検査せず**主張する（危険を宣言） | `assumeSorted`, `assumeUnique` |
| `enforce*` | 実行時に検査する | `enforce` |
| 長い名前 | コストの高い方・危険な方 | `byLine`（速い・エイリアス）/`byLineCopy`（安全・所有） |

遅延か即時かは性能だけでなく**正しさ**（元データの寿命、消費の一回性）の契約なので、
シグネチャを読まないと分からない API は事故を生む。「コストの高い方に長い名前」の非対称は、
既定の選択を正しい方向に倒す。
**理由**: 命名規約は「読まなくても当たるドキュメント」。1箇所でも破れると、
利用者は全 API で推測をやめてしまう。

> **読み替え注意**: 上の表は Phobos の体系の**実例**であり、移植すべきは字面でなく
> 「体系を1つ選び全 API で貫く」という原則。**対象言語に確立した体系があればそれが優先**
> — Rust なら動詞 (`split`) が遅延イテレータを返すのが標準（D と逆）で、危険の宣言は
> `assume*` でなく `unsafe fn` + `_unchecked`、実行時検査は `enforce` でなく `Result`。
> 自前の規約でエコシステムの規約を上書きすることこそ「規約を破る」行為になる。

## 原則3: 入口は少なく、語彙は広く

「少数の万能入口」（変換はすべて `to!T`）+「多数の直交する小さな動詞」（`retro`/`zip`/`chunks`…
合成で表現力が積になる）の二層にする。`fooWithSeparatorAndLimit` 型の複合関数を増やさない。
**理由**: API の学習コストは関数の数ではなく**「推測が外れる回数」**で決まる。
最悪なのは中途半端に重複した中粒度の関数群 — 利用者はどれを使うべきか毎回迷う。

## 原則4: 最頻の操作は儀式ゼロで動かす

デバッグ出力・ログ・文字列化のような1日数十回書かれる操作は、型もフォーマットも指定せず
書けること（`writeln(任意型...)`、`%s` 万能指定子）。
**「既定で全部動く → 制御したければ細かく指定できる」という勾配**が正しい。
逆（精密指定が必須）にすると、最も高頻度な場所に毎回の摩擦を置くことになる。

## 原則5: 満たした事前条件は型にして返す

「ソート済み範囲にしか二分探索してはいけない」をドキュメントでなく型で運ぶ。**3部品の分業**:

- 事前条件を**成立させ**、証拠の型を返す — `sort()` → `SortedRange`
- 証拠を**要求する** — `lowerBound` は `SortedRange` のメソッドとしてのみ存在
- 検査なしで証拠を**主張する** — `assumeSorted(外部由来データ)`（名前がリスクを宣言）

**理由**: ドキュメントの事前条件は読まれないか、リファクタリングで前提が崩れたとき沈黙する。
型で運べば `sort` の呼び忘れがコンパイルエラーになる。重要なのは、正しい使い方が
**最も短い書き方**でもあること — 利用者は安全のために余計なコードを書くのではなく、
自然に書くと安全になる。

## 原則6: 文法を持つ文字列はコンパイル時に検査する

フォーマット文字列・正規表現・パスパターン・数値リテラルのような「独自の文法を持つ
文字列」（いわゆる埋め込みミニ言語/DSL）は、ほぼ常にコンパイル時定数として書かれる
のだから、コンパイル時に構文と引数を突き合わせる
（`format!"%d"(3.14)` → コンパイルエラー）。追加の構文コストほぼゼロで、
「ログ出力など滅多に通らないパスに潜むフォーマット不一致」というバグクラスごと消える。
CTFE/constexpr/proc-macro を持つ言語でこれをやらない理由はない。

## 原則7: エラー処理は式の形で書かせる

try/catch は文なので、変数宣言と初期化を分断しスコープを深くする。
「失敗したら既定値」「例外を値として回収」「テストでの例外検証」のような頻出パターンを
式で書けるヘルパー（`to!int(s).ifThrown(0)`, `collectException`, `assertThrown`）にし、
手書き try/catch を本当に固有の回復処理がある場面だけに減らす。
（`Result`/`Option` + `?` 演算子の言語ではエラーが最初から式 — この原則は例外ベースの
言語でそれに近づけるための処方。）

## 原則8: 利用者が自然に書いた形を受け入れる

「正しい渡し方」を1つに強制しない。述語はラムダ・文字列・既存関数のどれでも、
出力先はバッファでもデリゲートでも、オプションは変数ポインタを並べるだけ（getopt —
受け手の**型がそのまま仕様になる**宣言的 API）。コンパイル時にも同じ API が動くこと
（実行時と別の constexpr 用語彙を作らない）。
**理由**: 受け口が硬い API は、呼び出しのたびに「ライブラリの流儀への変換コード」を
利用者に書かせる — 全利用者が払う税。受け入れの複雑さはライブラリ内部に**一度だけ**
実装すればよい（実装機構は design-by-introspection スキルのパターンD）。

## 原則9: 抽象の既定 + 具象化の明示的な出口

戻り値は型名ではなく契約で返す（Voldemort 型 / `impl Trait`）のを既定にしつつ、
「型を名指ししたい場面」（フィールド保存・仮想 API 境界・プラグイン境界）のための
**公式の出口**を必ず用意する — 型消去ラッパ（`inputRangeObject`）、全実体化（`.array`）、
参照ラッパ（`refRange`）。
**理由**: 出口がライブラリ公式にないと、利用者は `typeof()` の曲芸や危険なハックに流れる。
「抽象の既定 + 具象化の出口」は「遅延 `splitter` + 即時 `split`」と同じ構造の繰り返し。

## 原則10: 摩擦は長所の裏面 — 交換レートで評価する

内部設計の各長所は、利用者視点の何かを**支出**して買っている。設計レビューでは
「この長所の裏で利用者は何を払うか」を必ず問う:

| 長所 | 利用者が払う代償 |
|---|---|
| 遅延がデフォルト（読みやすさ=速さ） | 消費の一回性（2回 foreach で2回目は空、コンパイルは通る）、値コピーの曖昧さ、ラムダ捕捉の時差 |
| 契約で返す（実装の自由） | 名指し不能、エラー/デバッガに読めない型名 |
| 制約 = 仕様（呼び出し地点エラー） | 「どの条件が偽だったか」が読めないエラーメッセージの壁 |
| コンパイル時述語（インライン化） | 実行時に述語を選べない、インスタンス化の数だけバイナリ膨張 |
| 例外ベース（既定が簡潔） | nothrow/@nogc 文脈で使えない、高頻度エラーで重い |

代償を消せないなら、せめて**公式の緩和策**（複製版・型消去・実行時版）を併設し、
ドキュメントで正直に告知する。
なお「正しい既定が性能・正しさの両面で最善でない」と、知識のある利用者ほど
毎回オプトアウトを書く転倒が起きる（Phobos 自動デコードの教訓）—
**既定の選択こそ API 設計最大の決定**であり、後から変えられない。

## 原則11: 性能コストの配分を設計する

性能コストを**API のどちら側に置くか**は API 設計の決定であり、実装技法（performance
スキル）とは別の判断:

- **最適化はジェネリックな入口の裏に隠す**: 利用者に「速い専用関数」を覚えさせない。
  「ジェネリックだから遅い」を許すと利用者は専用関数に逃げ、ジェネリック API は信用を失う。
  ディスパッチはライブラリが一手に引き受ける。
- **前処理コストは利用者に選ばせる**: 前計算（検索パターンからスキップ表を作る等）を払えば
  後が速くなる最適化は、汎用 API に組み込まず別 API に。1回きりの利用では前処理が本体を
  上回る — ライブラリが勝手に「賢い」選択をすると、裏目に出る利用者から制御を奪う。
  既定は素直、前処理版はオプトイン。
- **高コストな既定動作には公式のオプトアウト**: 公式の逃げ道があれば利用者は安全に速くなり、
  なければ各自が危険なハックを再発明する。
- **sink（出力先）直接書き込みを基本にし、文字列を返す版はその上に作る**:
  「文字列を返す」設計は全呼び出しに割り当てを強制し後から取り除けない。
  「sink に書く」設計は割り当てを呼び出し側の選択にする。**向きは逆にできない**。
- **精度の段階を正式 API で公開**: 「精度を落とすと速くなる」選択肢（粗いクロック、
  非安定ソート）を型パラメータ等で公開する。

- [ ] 命名は体系の規約に従っているか（破ると利用者は全 API で推測をやめる）
- [ ] 最頻操作は無指定で動くか。既定→精密制御の勾配があるか
- [ ] 事前条件・状態遷移を型で表せないか（ドキュメント頼みになっていないか）
- [ ] 文字列引数が独自の文法を持っていないか（フォーマット・パターン等）。あればコンパイル時検査できないか
- [ ] 利用者の自然な書き方を何種類受け入れられるか。変換コードという税を課していないか
- [ ] 抽象で返すなら具象化の出口はあるか
