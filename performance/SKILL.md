# 性能設計

背骨は1つ:**最適化とは、型・入力サイズ・データの統計的構造ごとに最適な実装を
選び分けること**。単一の実装を全入力に適用しない。

横断する思考法は**3つの会計**: 割り当ての回数 / 同期の回数 / システムコールの回数。
いずれも「**回数 > 単価 > データ量**」の順に支配的で、最適化とはまず回数を数えて設計すること。

> **読み替え注意**: 原則の層（3つの会計、最弱の同期機構、メタデータ遅延）は言語非依存。
> 一方、**機構の層は D 正典** — コンパイル時ディスパッチ（`static if`）、
> CTFE テーブル/コード生成、`= void` 初期化スキップ、遅延レンジの合成融合は、
> 対象言語の等価機構（C++: `if constexpr`/constexpr、Rust: マクロ/const fn/単相化、
> Zig: comptime）に読み替える。**等価機構が無い言語（Go 等）ではその最適化は不可能**と
> 判断し、実行時ディスパッチや外部コード生成（同期ずれの代償つき）への後退を明示的に選ぶ。
> 無理に再現しない。

> **スコープ注意**: 性能コストを**公開 API のどちら側に置くか**（前処理のオプトイン、
> 高コスト既定動作のオプトアウト、sink 方向、「ジェネリック入口の裏に最適化を隠す」）は
> ライブラリ提供側の設計判断 — **api-ergonomics スキルの原則11** を参照。
> 本スキルは実装そのものの技法を扱う。

## 1. ディスパッチの階層 — どこで実装を選ぶか

- **型で**（コンパイル時）: 配列なら memcmp/memcpy 水準へ、1バイト要素のデフォルト述語検索は
  libc `memchr`（SIMD で素朴ループの1〜2桁速い）へ。**下層に存在する性能を上層で捨てない**。
  コンパイル時実行と両立させるには「CTFE 中か」で分岐し、純粋実装にフォールバック
  （D: `if (!__ctfe)`）— 1つの関数が両方の世界で最適実装を持てる。
- **入力サイズで**: 小入力は専用コード（再帰せず挿入ソートへ、5要素なら分岐構造を固定した
  手書き比較列＝最悪比較7回のソートネットワーク）。パラメータ自体もサイズで変える
  （ピボット候補のサンプル数: 512 要素未満は3点 / 以上は5点 — 大きい配列ほど悪い分割の
  代償が高いので、サンプルを増やす価値がある）。
- **データの統計的構造で**: 実世界の入力はランダムではない。テキストは ASCII が支配的
  → 重い汎用処理の前に「ASCII なら1命令で完了」の分岐を置く（平均コストが**少数派の頻度に
  比例**する構造になる）。配列は部分的にソート済みであることが多い → 既にソートされた
  連続区間を検出してそれをマージの単位にする（TimSort のラン検出。完全ソート済みなら O(n)）。
  さらに「マージ中に片側からの採用が続いたら、1要素ずつの比較をやめて一括コピーに切り替える」
  ようなヒューリスティクスは、効き具合に応じて**切り替え閾値自体を実行時に調整**できる
  （TimSort のギャロップ）。
- **入力の性質でエンジンごと自動選択**: 同じ問題に性格の違う複数エンジンがあるなら、
  入力を解析して自動で選ぶ。正規表現なら: 全候補を試すバックトラック型（後方参照まで
  表現できるが最悪指数時間 = ReDoS）と、状態集合を同時に進める Thompson NFA 型
  （表現力は落ちるが入力長×パターン長の線形保証）— 後方参照を使うパターンのときだけ
  前者に切り替える。**「保証の強いエンジンを既定に、表現力が必要なときだけ弱い保証へ」**
  — 呼び出し側に選ばせない。
- **最悪ケースを保証する**: 平均が速く最悪が壊滅的なものは、劣化を検知して堅実な実装へ
  フォールバックする保険をかける（introsort = 再帰深度予算切れで heapsort）。
  基盤コードは敵対的・病的入力にも使われる。

## 2. 割り当ての会計

- 変換パイプラインは遅延にして消費まで割り当てゼロ。ラッパーの合成には恒等式が成り立つ
  ことがあり（逆順の逆順 = 元のまま、間引きの間引き = 積の間引き）、それを型レベルで
  畳み込めば合成で生じる間接層も消える。
- **総量が事前に分かるか**を能力で静的判定（`hasLength`）→ 一度の確保 + 直接書き込み。
  割り当て1回・コピー0回に勝る成長戦略は存在しない。
- 成長戦略は対数スケーリング（小は約2倍、大きいほど成長率を下げる — 固定2倍は巨大バッファで
  最大50%浪費、固定1.5倍は小バッファで再割り当て過多）。再割り当ての前に
  **in-place 拡張（realloc/GC.extend）を試す** — 成功すれば全コピーが消える。
- すぐ上書きするものをデフォルト初期化しない。ただし省ける範囲は型の性質
  （ポインタを含むか = GC 誤スキャン・安全境界）で精密に判定する。
- 高頻度・短寿命の呼び出し（マッチング、リクエスト処理）は呼び出しごとの割り当てゼロを
  設計目標に: 必要最大量を事前計算して一括確保し、next ポインタを要素自身に埋め込んだ
  フリーリストで O(1) に貸し借りする。可変長の結果は「典型サイズまでは構造体内に
  インライン格納、超えたときだけヒープ + 参照カウントで共有し書き込み時にコピー」
  (SSO + CoW) の二段構えに。
- 境界をまたぐ小さな変換（C 文字列化など）はスタックバッファ + 溢れたときだけヒープ。
  回数が多い場所ほど塵が積もる。

## 3. データレイアウトとテーブル

- 疎で巨大なキー空間（Unicode の21ビット空間等）の表は、キーをビット区間で分割した
  多段テーブル（Trie）にし、同一内容のサブテーブルを共有して圧縮。参照は表引き3回に
  増えるが全段が L1/L2 に収まる —「キャッシュヒット3回」は「キャッシュミス1回」より速い。
- 表はビットパックし（シフト+マスク ≪ キャッシュミス）、生成はコンパイル時計算で
  ソースに焼き込む — **生成コードと生成物が同じソースにある**ので外部生成スクリプトの
  同期ずれ問題ごと消える。
- ビット列はワード単位で処理。**データ表現の選択が使える命令（popcnt/bswap/SIMD）を決める**。
- VM のバイトコードは固定幅整数にパック（歩進が予測可能、IR がキャッシュに密に載る）。

## 4. CPU を遊ばせない

- **依存チェーンを分割**: 高レイテンシ演算（FP 加算 3〜5 サイクル）の畳み込みを単一
  アキュムレータに直列依存させない。複数アキュムレータ or ペアワイズ木 — 後者は丸め誤差も
  O(n)→O(log n) に改善する（**正確さのための変更が同時に速さになる**例）。
- 回数がコンパイル時に分かる小ループは展開（ループ制御が本体より高い）。
- 型に付随する高コスト操作（コピーフック・デストラクタ・初期化）は、観測可能な意味論を
  保ったまま省略できる局面を型の性質で静的に見つけて省く（swap の生バイト3コピー）。
- 並列化は「スレッド / ILP / SIMD」の3層。**スレッドを増やす前に1コアを使い切る**
  （スレッド内でもアキュムレータ分割）。

## 5. 同期の会計

- 機構には強さの階層がある: なし < TLS < atomic < spinlock < mutex+condvar。
  守りたい不変条件を言語化し、**それを守れる最弱の機構**を選ぶ。1段強すぎる選択は
  そのままスループット低下。
- **同期の設計とはまず同期の回数の設計**: 均等分割の分配はアトミック加算1命令で足りる
  （キューは過剰装備）。キューが要る場合も投入はバッチで O(タスク数)→O(1) ロックに。
  ロックは操作単位で1回取り、内側ではロックなし版を使う（stdio の flockfile +
  `fputc_unlocked`、1回の出力呼び出しでロック1往復）。
- **偽共有はレイアウトで予防**: スレッドごとデータをキャッシュライン整列 + **前後1ライン
  パディング**（隣接ヒープ割り当てとの偽共有まで防ぐ）。プロファイラに現れにくいので
  事後検出でなく予防が唯一の解。
- ワークユニット数はスレッド数×4程度を入力長から導出（大きすぎ→負荷不均衡、
  小さすぎ→同期コスト過多）。※ CPU バウンドのデータ並列が前提 — I/O バウンドでは
  コア数より多くの並行度が効く。
- パイプラインの接続はバッファの中身を詰め替えず**ポインタ/スライスの交換**で。

## 6. システムコールの会計

- **発行する syscall の本数を数えて設計する**。機能検出付きフォールバック階段の形になる
  （FD クローズ: `closefrom` 1回 → `/dev/fd` 列挙で開いている分だけ → `poll` 判定 →
  最後の手段の総当たり）。
- メタデータは参照されるまで遅延し、取得したらフラグ付きでキャッシュして二度呼ばない
  （`readdir` が無料でくれる情報で足りる判定はそれで済ます）。
- サイズを聞けるなら聞いてから1回で確保。確保したメモリの性質をランタイムに教える
  （ポインタ非含有なら GC スキャン除外）。ただしサイズは TOCTOU 的に信用せず
  read ループは部分読みと EINTR を処理する。
- **OS/libc に既にある最速プリミティブを探してから書く**（行読みは `getdelim`)。
  存在をコンパイル時/実行時に検査し、なければ自前実装へフォールバック。
- mmap の窓掛けは前後を重ねて張る（ヒステリシス）— mmap/munmap 自体も安くない。
- 時刻は「フィールドの集合」でなく数直線上の整数1本で持つ（比較・差分が1命令、
  タイムゾーンは表示時のみ）。暦変換は周期性（400年=146097日)を使った階層除算+表引き。
- 現在時刻の取得もコストは一様ではない。ログのタイムスタンプやタイムアウト判定に
  ナノ秒精度は要らない — 精度を落とした分だけ安いクロックを明示的に選ぶ。
  Linux なら `clock_gettime` の `CLOCK_REALTIME_COARSE` / `CLOCK_MONOTONIC_COARSE`:
  タイマー割り込み粒度（数 ms）の値を vDSO 経由（カーネルに入らず共有メモリページを
  読むだけ）で返すため、高精度版よりずっと安い。要求精度を API のパラメータで受けて
  クロックに振り分けると、利用者ごとに最適点を選べる。

## 7. インタプリタ/VM の技法

- 命令列が不変なら、中心ループを毎ステップの巨大 switch にせず**実行前に解決した
  関数ポインタ表**で直線化。各ハンドラは命令種別ごとに特殊化して内部分岐を消す。
- 並行に試行する実行パス（NFA の状態、探索の枝）が指数的に増えうるなら、
  「同じ位置・同じ状態」に到達したパスを検出して1つに合流させる — これが線形時間保証の
  実体で、合流がなければ `(a|a)*` 型の入力で爆発する。合流済み判定のテーブルは
  毎ステップ memset でクリアせず、世代カウンタ（「最後に通過した世代」を記録し
  「今の世代か」を比較）にすればクリアが O(0) になる — テーブル再利用全般に効く汎用技。
- マッチ実時間の大半は「マッチしない場所の読み飛ばし」。パターン先頭 N 文字分の一致状態を
  1ワードのビット列に詰め、1文字ごとにシフト+OR だけ（分岐なし）で更新するプリフィルタ
  （Shift-Or）で候補位置まで飛び、先頭文字が固定なら memchr とさらに多段に組む。
- 実行前に IR へ覗き穴最適化を1周（ループ脱出判定に文字集合 Bloom フィルタを焼き込む等）。
  毎回払うコストを前処理1回の解析で買い取る。
- **究極の特殊化はコンパイル時コード生成**: パターンが静的に既知なら、インタプリタでなく
  入力専用の制御フローを生成してコンパイラに渡す（ctRegex）。代償はコンパイル時間。

## 規律と落とし穴

- **ランダム入力のベンチマークだけで測らない**。実データの分布（ASCII 率、部分ソート率、
  キャプチャ数の典型値）こそ漸近計算量に現れない最重要の最適化資源であり、
  構造活用型はランダム入力では差が出ない。
- ライブラリとして公開するコードでは、最適化を API 表面に出すか裏に隠すかの判断が
  別途必要 — api-ergonomics スキルの原則11を参照。
