# skills

## インストール

```sh
ln -s /path/to/this/repo ~/.claude/skills
```

スキルはセッション開始時に読み込まれる。

## スキル一覧

| スキル | いつ使うか | 概要 |
|---|---|---|
| [design-by-introspection](design-by-introspection/SKILL.md) | ジェネリックな部品（コンテナ・range・アロケータ・シリアライザ・フック/ポリシー型）を設計するとき。継承やインターフェース爆発を解きほぐすとき | 型の能力をコンパイル時に観測して構造・API・実装を組み替える設計手法。「必須は最小、任意は内省で発見、なければ既定か不在」。パターンカタログ A〜H と落とし穴。**意図的に D を正典**とし、他言語への可否判断表を持つ |
| [api-ergonomics](api-ergonomics/SKILL.md) | ライブラリ・ツールの公開 API（命名、引数、戻り値、エラー、性能コストの配分）を設計・レビューするとき | 判断基準は「呼び出し地点だけ見てコードレビューができるか」。命名規約の体系、事前条件を型で返す、抽象の既定+具象化の出口、摩擦は長所の裏面（交換レート評価）、性能コストをAPIのどちら側に置くか、の11原則+チェックリスト |
| [performance](performance/SKILL.md) | ホットパス・基盤コードの性能設計と最適化。割り当て/ロック/システムコールを減らしたいとき、パーサ・VM・並列処理・I/O の実装時 | 背骨は「型・入力サイズ・データの統計的構造ごとに実装を選び分ける」。横断する思考法は3つの会計（割り当て・同期・syscall、いずれも回数 > 単価 > データ量）。実装技法に限定し、API 表面の判断は api-ergonomics に委譲 |

## スキル間の関係

```
api-ergonomics ──(公開 API の判断: コストをどちら側に置くか)
      │ 原則8 の実装機構             │ 原則11 ⇄ 実装技法
      ▼                             ▼
design-by-introspection         performance
（型能力で実装を組み替える機構）   （実装そのものの技法）
```

## 執筆方針

- **抽象概念が本文、固有名詞は括弧内のポインタ**。仕組みと教訓を自己完結に書き、
  TimSort や Thompson NFA 等の名前は索引タグとして添えるだけにする
  （固有名詞を知らない読者でも教訓が再構成できることをテスト済み）
- **言語非依存の原則と D 固有の機構を分離**し、機構には「読み替え注意」
  （他言語の等価機構、無い場合は不可能と判断する旨）を付ける
- **スキル名は実際のスコープと一致させる**。汎用知識とライブラリ提供側固有の知識は
  スキルを分けて相互参照する
- 編集時は検証シナリオ（ベースライン採取 → 執筆 → サブエージェントでの retrieval/適用テスト）を回す
